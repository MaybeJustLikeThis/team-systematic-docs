# 07. 质量、评测、指标和度量

## 目标

企业级 AI 开发必须证明质量，而不是相信 AI 的自我描述。

质量体系分为四层：

```text
代码质量
测试质量
协作质量
业务质量
```

## 代码质量

检查项：

- 格式化。
- Lint。
- 类型检查。
- 复杂度。
- 文件大小。
- 重复代码。
- 依赖新增。
- 边界条件。
- 错误处理。

AI 常见风险：

- 写出看似合理但不符合项目边界的代码。
- 引入无关重构。
- 过度抽象。
- 漏掉错误路径。
- 生成未使用代码。
- 改变隐式兼容行为。

## 测试质量

测试必须覆盖：

- 核心路径。
- 错误路径。
- 边界条件。
- 权限场景。
- 数据为空。
- 数据异常。
- 并发或重试。
- 向后兼容。

AI 生成测试的审查重点：

- 是否真的验证行为，而不是只验证 mock 被调用。
- 是否覆盖失败路径。
- 是否对历史 bug 有回归测试。
- 是否因为实现细节变化而脆弱。

## 协作质量

度量：

- PR 是否包含 AI Trace。
- PR 是否声明 Memory Used。
- PR 是否声明 Memory Impact。
- 是否有 human owner。
- 是否有风险等级。
- 是否有测试证据。
- 是否有 reviewer 发现问题。

## 业务质量

AI 代码最终要服务业务目标。

检查：

- 是否解决原始问题。
- 是否符合用户路径。
- 是否影响关键指标。
- 是否破坏旧用户行为。
- 是否需要灰度。
- 是否有回滚方案。

## AI 协作指标

### 效率指标

- 需求到 PR 时间。
- PR 周期时间。
- 测试补充时间。
- 文档更新时间。
- CI 修复时间。

### 质量指标

- AI PR 缺陷率。
- Review 发现问题数。
- 线上回归数。
- CI 首次通过率。
- 测试覆盖变化。

### 治理指标

- AI Trace 覆盖率。
- Memory 引用率。
- Memory 候选采纳率。
- Hook 命中率。
- Blocking Hook 误报率。
- 高风险任务审批覆盖率。

### 知识增长指标

- 新增 Memory 数。
- deprecated Memory 数。
- pitfall 转 Hook 数。
- 事故复盘 Memory 数。
- 高频引用 Memory Top N。

## AI Review Checklist

```text
1. 这个变更是否真的解决需求？
2. 是否修改了无关文件？
3. 是否违反已有 Memory？
4. 是否引入新依赖？
5. 是否改变公开接口？
6. 是否遗漏错误处理？
7. 是否有足够测试？
8. 是否涉及敏感数据？
9. 是否需要人工审批？
10. 是否需要更新文档或 Memory？
```

## 评测机制

### Prompt / Agent 评测

为常见任务建立 golden set：

- 修 bug。
- 补测试。
- 写 PR 摘要。
- 做 code review。
- 分析 CI 失败。
- 提取 Memory。

每类任务保留标准输入和期望输出，用于评估不同 Agent 或模型版本。

### 输出评分

建议维度：

| 维度 | 说明 |
|---|---|
| 正确性 | 是否解决问题 |
| 完整性 | 是否覆盖必要步骤 |
| 可维护性 | 是否符合项目风格 |
| 安全性 | 是否引入风险 |
| 可验证性 | 是否有测试和证据 |
| 可追踪性 | 是否引用 Memory 和 artifact |

## 完成定义

AI 参与的任务只有同时满足以下条件，才算完成：

- 验收标准通过。
- 测试或验证证据存在。
- PR review 完成。
- 风险等级正确。
- 高风险审批完成。
- AI Trace 记录存在。
- Memory Impact 已处理。
- 没有阻断级 finding。
